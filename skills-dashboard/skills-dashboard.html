没找到匹配的技能，试试换个关键词？

一个 Skill 就是一份「专属说明书」，装好后 AI 在恰当的时机自动翻阅，照着里面的规范、流程、工具来做事。

每个 Skill 由 7 个维度 定义自己——缺一不可，写满才合格：

1
元数据（我是谁）
名字 + 触发条件 + 排除场景，告诉 AI 什么时候翻这份说明书
2
边界（管什么 / 不管什么）
划定能力范围，明确前置条件，不越界不乱干
3
流程（分几步干）
每一步：输入什么 → 做什么 → 输出什么 → 下一步去哪
4
工具（靠什么干活）
脚本、模板、调色板、命令行工具、API 等具体依赖
5
容错（出错了怎么办）
每种可能的失败都有 B 方案或 fallback，不崩溃不沉默
6
评测（怎么算干好）
可验证的是非题——"标题字体是 Poppins 吗？" 而不是 "好看吗？"
7
验证（怎么证明正确）
事后检查——输出能正常打开吗？结果可复现吗？各项指标达标吗？

动手教学：手把手拆解 brand-guidelines

下面是 brand-guidelines（品牌配色规范）的完整 SKILL.md 写法，逐维度标注。读完你就能照着写自己的。

SKILL.md 完整源码
▼ 维度 1：元数据 — 写在文件最顶部的 YAML 头（frontmatter）
---
name: brand-guidelines          ← 唯一 ID，英文小写，连字符分隔
description: "Applies Anthropic's official brand colors and typography..."
                    ← 最重要！一句话说清：干什么 + 什么时候触发
                      AI 靠这个判断"我该不该翻这本说明书"
---
🧑 人话翻译：--- 是 YAML 格式的开场标记，写技能文件必须用它开头和收尾。下面两行是真正的内容：

name: brand-guidelines → 这技能的英文名叫 brand-guidelines（品牌指导规范），全部小写，单词中间用短横线连起来，不能有空格。

description: "Applies Anthropic's official brand colors and typography..." → 这句英文直译是「把 Anthropic 公司官方的品牌色和字体套到任何作品上」。注意这句描述里藏了两个信息：它能干什么（套品牌色和字体）和什么时候触发（当用户提到品牌色、字体、风格规范的时候）。AI 就是靠读这句话判断"我该不该翻这本说明书"。

最后再写一个 --- 表示 YAML 头结束。
▼ 维度 2：边界 — 第一段概述就划清"管什么、不管什么"
# Brand Guidelines

This skill applies Anthropic's brand colors and typography to any artifact.
← 管什么：任何类型的作品（PPT/文档/网页/海报...）

It covers colors and fonts only. It does NOT handle layout,
animation, illustration style, or voice & tone.
← 不管什么：布局 / 动画 / 插画风格 / 文案语调 —— 明确排除

Prerequisites: none. Works with any output format (HTML, PPTX, PNG, etc.).
← 前置条件：无，纯色值+字体方案，不依赖任何外部工具
🧑 人话翻译：# Brand Guidelines → 这是正文的大标题，直译就是「品牌指导规范」。

第一句 This skill applies Anthropic's brand colors and typography to any artifact. → 直译：这个技能把 Anthropic 的品牌颜色和字体应用到 任何作品 上。注意 any artifact 这个词——不管是 PPT、文档、网页、海报，通通都能用。

第二句 It covers colors and fonts only. It does NOT handle layout, animation, illustration style, or voice & tone. → 直译：这个技能 只管颜色和字体，不管布局、不管动画、不管插画风格、不管文案语气。这句就是画边界线——线内的我才管，线外的别找我。

第三句 Prerequisites: none. Works with any output format. → 直译：没有前置条件（不用装任何软件），输出什么格式都行（HTML、PPTX、PNG 随便）。
▼ 维度 3：流程 — 分步执行，每一步有判断标准
## Execution Steps

Step 1: Determine the artifact type (slide, doc, web page, etc.)
  → If unknown → ask user "What format are you outputting to?"
  ← 有判断分支：不知道就问，别猜

Step 2: Select color roles: headings → #141413, CTAs → #f59e0b, ...
  → If high-contrast context → use white text on dark background instead

Step 3: Apply fonts: Heading=Poppins, Body=Lora
  → If output is plain text → skip fonts, use system default

Step 4: Run fallback check: are all specified fonts available?
  → If any missing → activate fallback chain automatically
🧑 人话翻译：## Execution Steps → 二级标题，直译「执行步骤」。下面写了 4 步，每一步都是"干什么 → 然后怎么判断"的结构：

Step 1: Determine the artifact type → 先搞清楚你要做的作品是什么类型——是幻灯片、是文档、还是网页？
　→ If unknown → ask user "What format are you outputting to?" → 如果 AI 判断不出来是什么类型，就直接开口问用户「你要输出什么格式？」——别自己瞎猜。

Step 2: Select color roles → 给不同文字指定颜色：标题用主色 #141413，按钮和链接用辅助色 #f59e0b。
　→ If high-contrast context → use white text on dark background → 如果用户要的是深色背景，那字就不能用深色了，切换成白字。

Step 3: Apply fonts → 应用字体：标题用 Poppins，正文用 Lora。
　→ If output is plain text → skip fonts, use system default → 但如果输出的是纯文本（没有任何格式那种），那就别管字体了，用系统默认字体就行。

Step 4: Run fallback check → 最后检查一遍：指定的字体系统里都有吗？
　→ If any missing → activate fallback chain automatically → 如果哪个字体不存在，自动启动降级链条。
▼ 维度 4：工具 — 列出所有具体的"生产资料"（色板 + 字体表）
## Color Palette

| Role        | Hex       | Usage                     |
|-------------|-----------|---------------------------|
| Primary     | #141413   | Headings, body text       |
| Secondary   | #f59e0b   | CTAs, hover states        |
| ...         | ...       | ...                       |

← 色值精确到 hex——不给 AI 模糊空间，"深灰"不靠谱，#141413 才靠谱

## Typography

| Role    | Font Stack                        |
|---------|-----------------------------------|
| Heading | Poppins, sans-serif               |
| Body    | Lora, Georgia, serif              |

← 字体栈 = 备胎链：Poppins 没有→Arial，Lora 没有→Georgia→系统字体
🧑 人话翻译：这里两张表，就是给 AI 的"原料清单"。

## Color Palette（调色板）：三列分别是 Role（用途）、Hex（色值）、Usage（用在哪）。比如主色 #141413 用于标题和正文，辅助色 #f59e0b 用于按钮和鼠标悬停。每一行都是一个精确的"什么颜色往哪招呼"的指令。

## Typography（字体排印）：两列，Role（用途）、Font Stack（字体栈）。标题用 Poppins，正文用 Lora。"字体栈"就是一串备胎：Poppins 没装 → 用 sans-serif 兜底；Lora 没装 → 用 Georgia，再没有 → 用系统 serif。

这两张表的核心思路一样：只给精确数据，不给模糊描述。
▼ 维度 5：容错 — 每种可能翻车的场景，都有明确的"那就……"
## Fallback Strategy
- Poppins unavailable → fall back to Arial
- Primary color contrast fails → switch to black text
← 每一行一个"万一……那就……"
🧑 人话翻译：## Fallback Strategy → 二级标题，直译「降级策略」——其实就是"万一出状况了怎么办"。

第一条 Poppins unavailable → fall back to Arial → 万一 Poppins 字体在系统里没装，那就改用 Arial。
第二条 Primary color contrast fails → switch to black text → 万一主色文字在背景上对比度不够（看不清），就直接切回黑色。

容错策略的写法套路就是"万一……那就……"，一两行就够。你不写的话，AI 遇到意外会自己猜一个方案，那个方案大概率不是你想要的。
▼ 维度 6：评测 — 干完活立刻自查，每条都能答"是/否"
## Verification Checklist
- [ ] Are all headings in Poppins?    ← 是/否问题，不模糊
- [ ] Are all CTAs using #f59e0b?
- [ ] Did any font fallback activate? If yes → is the result still readable?
- [ ] Do colors pass WCAG AA contrast ratio?
🧑 人话翻译：## Verification Checklist → 直译「验证清单」，是干完活立刻自检的列表。每行 - [ ] 是一个可勾选条目：

第一条：所有标题字体是 Poppins 吗？
第二条：所有按钮颜色是 #f59e0b 吗？
第三条：触发字体降级了吗？如果触发了，文字还能看清吗？
第四条：颜色搭配通过了 WCAG AA 无障碍对比度标准吗？

关键：每条都是"是/否"问题，不是"看起来好看吗"这种没法客观判断的东西。
▼ 维度 7：验证 — 事后确认"输出能不能正常用"
## Post-Output Validation
- Output file opens correctly in target application
- Brand colors render identically across Chrome / Safari / Firefox
- Text contrast ≥ 4.5:1 when measured with a contrast checker
🧑 人话翻译：## Post-Output Validation → 直译「输出后验证」，文件生成之后才做的最终确认，跟评测不一样——评测是自己查自己的，验证是查"这东西能用吗"：

第一条：输出的文件在目标程序里能正常打开不报错吗？
第二条：品牌色在 Chrome、Safari、Firefox 三个浏览器里颜色一样吗？
第三条：用工具量一下文字对比度≥4.5:1 了吗？

评测（维度6）和验证（维度7）的区别：评测是"我按规范做了吗"，验证是"做出来的东西能用吗"。一个问过程，一个问结果。

## 执行步骤（维度 3：流程）

  输入 → 处理 → 输出 → 判断（如果 A 则 B）
### Step 2：XXX
  ...

## 工具清单（维度 4）
列出所有要用的数据、模板、脚本、API
精确到文件名 / 色值 / 命令 / URL

## 异常处理（维度 5：容错）
- 如果 A 失败 → 走 B 方案
- 如果 C 不存在 → 提示用户 XXX

## 评测检查（维度 6：评测）
- [ ] 输出是 XXX 格式吗？
- [ ] 关键字段都有值吗？
- [ ] 颜色/字体/格式符合规范吗？

## 最终验证（维度 7：验证）
- 输出文件能被目标程序正常打开吗？
- 在常见环境下（浏览器/系统）表现一致吗？
- 如果输入相同，输出可复现吗？
# 维度 核心问题 写在 SKILL.md 的哪里

2边界管什么？不管什么？需要什么前置条件？正文第一段（概述 / 范围）
3流程分几步？每步：输入→处理→输出→判断？## Execution Steps / ## 执行步骤
4工具靠什么干活？精确到文件名、色值、URL？## Color Palette / ## Tools / 数据表
5容错每种失败模式都有 B 方案吗？## Fallback / ## 异常处理 / 流程中的 if→else
6评测怎么证明干好了？能写成是/非检查项吗？## Verification Checklist
7验证输出能正常用吗？可复现吗？跨环境一致吗？## Post-Output Validation / ## 最终验证

自己创建新技能

7 步检查清单 + 可复制模板

  • 想好名字和触发条件（description 最最重要）
  • 明确边界：做什么、不做什么、需要什么环境
  • 写出分步执行流程（每步输入 → 输出 → 判断）
  • 列出依赖的工具和脚本
  • 想好出错时的备用方案（fallback）
  • 写 3~5 个可验证的测试断言
  • 补充排除条件（SKIP），避免误触发
